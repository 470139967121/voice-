# ตัวรันการทดสอบเส้นทางผู้ใช้บนอุปกรณ์ของ ShyTalk

_นี่คือคำแปลของ JOURNEY-RUNNER.md_

`device-journey-runner.js` ขับเคลื่อน **แอป ShyTalk จริงบนโทรศัพท์ที่เชื่อมต่ออยู่**
ผ่านเส้นทางผู้ใช้แบบครบวงจร และเขียน **รายงานผ่าน/ไม่ผ่านอย่างละเอียด** ที่คุณ
สามารถอ่านได้ — ดังนั้นคุณรันคำสั่งเดียวและอ่านรายงานเดียวแทนที่จะแตะ
ผ่านทุกขั้นตอนด้วยตนเอง

มันเป็นตัวรันแบบ **ไฮบริด** แต่ละเส้นทางสามารถตรวจสอบสามชั้นพร้อมกันได้:

1. **UI** — แตะ/ตรวจสอบแอปที่ทำงานอยู่จริงผ่าน `adb` + `uiautomator` (`testTag` ของ Compose
   ปรากฏเป็น `resource-id` ในดัมป์; ไดอะล็อกถูกจับคู่ด้วย
   ข้อความที่มองเห็นได้)
2. **Firestore** — อ่านอีมูเลเตอร์ในเครื่องโดยตรง (ผ่าน `firebase-admin`) เพื่อ
   ยืนยันสถานะฐานข้อมูลที่อยู่เบื้องหลังแต่ละการกระทำ
3. **เซิร์ฟเวอร์ / API** — ลงชื่อเข้าใช้ในฐานะแต่ละเพอร์โซนา (โทเค็น ID จริงของ Firebase จาก
   อีมูเลเตอร์ Auth) และเรียก `express-api` ดังนั้นจึงตรวจสอบ **กฎที่
   เซิร์ฟเวอร์บังคับใช้** (ด่านกั้น cohort ของ OSA, การแทนที่ของผู้ดูแลระบบ, การกลั่นกรอง) — ซึ่ง
   _ไม่_ ปรากฏใน UI เพียงอย่างเดียว

> คำแปลของคู่มือนี้อยู่ใน `journey-runner-locales/` (20 ภาษา)

## 1. ข้อกำหนดเบื้องต้น

| สิ่งที่คุณต้องมี                 | อย่างไร                                                                                                                                      |
| -------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| **Docker Desktop** ที่กำลังทำงาน | สำหรับอีมูเลเตอร์ Firebase + LiveKit/MinIO                                                                                                   |
| **สแตกในเครื่องที่เปิดอยู่**     | `bash local/start.sh` (จากรากของรีโป) — เริ่มอีมูเลเตอร์ Firebase + express-api ปล่อยให้มันทำงานต่อไป                                        |
| **เพอร์โซนาที่ถูกซีดแล้ว**       | `cd express-api && node --env-file=.env.local scripts/seed-personas-local.js` (idempotent; ซีดชุดทดสอบ P‑02…P‑19 ด้วยรหัสผ่าน `localdev123`) |
| **โทรศัพท์ที่เชื่อมต่ออยู่**     | `adb devices` ต้องแสดงรายการหนึ่งเครื่อง (สาย USB **หรือ** `adb` ไร้สาย) อีมูเลเตอร์ Android ก็ใช้ได้เช่นกัน                                 |
| **Java 21+ และ Android SDK**     | จำเป็นเฉพาะครั้งแรกเท่านั้น เพื่อให้ตัวรันสามารถบิลด์แอปได้หาก APK หายไป                                                                     |

ตัวรันบิลด์ APK ดีบัก `local` ด้วยตัวเองหากยังไม่ได้บิลด์

## 2. รันมัน

จากรากของรีโป:

```sh
# Run the whole suite against the local stack
node express-api/scripts/device-journey-runner.js

# See the list of journeys without running anything
node express-api/scripts/device-journey-runner.js --list

# Run only specific journeys
node express-api/scripts/device-journey-runner.js --journeys J02,J08,J11

# Force a fresh APK build first
node express-api/scripts/device-journey-runner.js --rebuild

# Full option list
node express-api/scripts/device-journey-runner.js --help
```

ตัวเลือก: `--target local|dev` (ค่าเริ่มต้น `local`) · `--serial <adb-serial>`
(ค่าเริ่มต้น: เลือกอัตโนมัติ) · `--journeys <ids>` · `--rebuild` · `--no-reset` (ข้าม
การติดตั้งใหม่แบบสะอาดในเส้นทาง smoke) · `--out <dir>` · `--list` · `--help`

ตัวรันตรึง adb serial **หนึ่งตัว** สำหรับทุกคำสั่ง ดังนั้นมันจึงทำงานได้แม้เมื่อ
โทรศัพท์ปรากฏสองครั้ง (USB + ไร้สาย) สำหรับเป้าหมาย `local` มันตั้งค่า
อุโมงค์ `adb reverse` เพื่อให้แอปบนอุปกรณ์เข้าถึงสแตกบนเครื่องของคุณได้

เมื่อเสร็จสิ้น มันพิมพ์สรุปและเขียน ภายใต้ `journey-results/`:

| ไฟล์                            | คืออะไร                                                                                                             |

| `latest-report.md`              | **อ่านอันนี้** — แยกตามเส้นทาง แยกตามขั้นตอน ✅/❌ พร้อมเหตุผล, testTags บนหน้าจอ และลิงก์ภาพหน้าจอสำหรับทุกขั้นตอน |

| `runs/<runId>/*.png`            | ภาพหน้าจอของทุกขั้นตอน (ทั้งผ่าน _และ_ ไม่ผ่าน)                                                                     |
| `runs/<runId>/report.{md,json}` | รายงานที่เก็บถาวรสำหรับการรันนั้นโดยเฉพาะ                                                                           |

โค้ดทางออกเป็น `0` เมื่อทุกเส้นทางผ่าน และ `1` เมื่อมีเส้นทางใดไม่ผ่าน เมื่อเกิดความล้มเหลว
ขั้นตอนจะบันทึกสิ่งที่อยู่บนหน้าจออย่างแม่นยำ ดังนั้นคุณจึงเห็น _ว่าทำไม_ ได้โดยไม่ต้อง
ขับเคลื่อนโทรศัพท์ใหม่อีกครั้ง

## 4. สิ่งที่เส้นทางครอบคลุม

รัน `--list` เพื่อดูชุดที่ใช้งานจริง โดยสรุปแล้วชุดทดสอบครอบคลุม:

- **Smoke** — ติดตั้งใหม่แบบสะอาด → ยอมรับข้อกำหนดทางกฎหมาย → ลงชื่อเข้าใช้, แบ็กเอนด์เข้าถึงได้
- **การลงชื่อเข้าใช้ตาม cohort** — เพอร์โซนาผู้ใหญ่ / ผู้เยาว์ / ผู้ดูแลระบบ ลงชื่อเข้าใช้ผ่าน
  ตัวเลือกเพอร์โซนาแบบ dev ในแอป; ตัวตนได้รับการยืนยันเทียบกับโอเวอร์เลย์ดีบักและ
  ฟิลด์ `cohort` ของ Firestore
- **ด่านกั้น cohort ของ OSA** — ผู้เยาว์ไม่สามารถติดตามหรือดูผู้ใหญ่ได้ (เซิร์ฟเวอร์ส่งคืน
  `404` และการเขียนไปยัง Firestore ไม่เคยเกิดขึ้น) ในขณะที่การกระทำภายใน cohort เดียวกัน
  สำเร็จ — พิสูจน์ว่าด่านกั้นเจาะจงตาม cohort ไม่ใช่การบล็อกแบบเหมารวม
- **ผู้ดูแลระบบ** — การแทนที่ cohort สงวนไว้สำหรับเจ้าหน้าที่เท่านั้น (สมาชิกทั่วไปถูกปฏิเสธด้วย
  `422`; บัญชีเจ้าหน้าที่สำเร็จและเขียนแถวการตรวจสอบตามข้อกำหนด)
- **การกลั่นกรอง** — รายงาน → ผู้ดูแลระบบระงับ (+ การตรวจสอบ) → อุทธรณ์ → ยกเลิกการระงับ ซึ่งทั้งหมด
  ถูกบังคับใช้โดยเซิร์ฟเวอร์ พร้อมการล้างข้อมูลแบบ idempotent

การยืนยันตัวตนในเส้นทางใช้ **ตัวเลือกเพอร์โซนาแบบ dev ในแอป** เสมอ — ไม่เคยใช้
การลงชื่อเข้าใช้ Google/Apple จริง

> **หมายเหตุเกี่ยวกับสเปกของเส้นทาง** แผน Gherkin ใน
> `journey-tests/j01-j19` เป็น _ภาพในอุดมคติ_ บางส่วน: พวกมันอ้างอิงถึง
> UI ที่แอปที่เผยแพร่ไม่มี (เช่น หน้าจอสมัครสมาชิกด้วยอีเมล/รหัสผ่าน, แท็บผู้เยาว์ที่ซ่อนอยู่,
> หน้าจอการค้นพบ) ดังนั้นตัวรันจึงแมปเจตนาที่แท้จริงของแต่ละเส้นทาง
> เทียบกับแอป **จริง** + Firestore + API และบันทึก
> ความแตกต่างเช่นนี้เป็นข้อค้นพบแทนที่จะล้มเหลวบนเรื่องที่ไม่มีจริง

## 5. การแก้ไขปัญหา

| อาการ                                                 | การแก้ไข                                                                                                                |
| ----------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| `No adb device found`                                 | เสียบ / จับคู่โทรศัพท์; ตรวจสอบ `adb devices`                                                                           |
| ค้างที่การเข้าถึง SignIn / "backend NOT reachable"    | สแตกในเครื่องไม่ได้เปิดอยู่ หรืออุโมงค์ `adb reverse` ไม่ได้ตั้งค่า — เริ่มต้น `bash local/start.sh` ใหม่และรันอีกครั้ง |
| `persona "<email>" not found in picker`               | เพอร์โซนายังไม่ถูกซีด — รันคำสั่ง seed ใน §1                                                                            |
| ไม่มี `Firestore assertions: ON` / ขั้นตอน DB ถูกข้าม | การตรวจสอบ DB จะรันเฉพาะสำหรับ `--target local` เท่านั้น                                                                |
| การบิลด์ APK ล้มเหลว                                  | เปิด `gradle-build.log` ที่พิมพ์ออกมา; ตรวจสอบให้แน่ใจว่าติดตั้ง Java 21+ และ Android SDK แล้ว                          |
| ขั้นตอนล้มเหลวบนหน้าจอที่คุณไม่ได้คาดไว้              | เปิดภาพหน้าจอที่ระบุชื่อใน `latest-report.md` สำหรับขั้นตอนนั้น                                                         |

## 6. การเพิ่มเส้นทาง

เส้นทางเป็นอ็อบเจ็กต์ธรรมดาที่มีเมธอด `run(device, reporter, ctx)` ประกอบขึ้น
จากตัวช่วยที่ใช้ร่วมกัน:

- `signInAs(device, reporter, ctx, email, nameToken)` — ลงชื่อเข้าใช้เพอร์โซนาผ่าน
  ตัวเลือกและผ่านหน้าจอคั่นของการเปิดครั้งแรกไปจนถึง Home
- UI: `tapId` / `waitForId` / `waitForText` / `selectPersonaByText` /
  `tapLowestText`, และ `dump(device)` + `byId` / `byText` / `byTextContains`
- Firestore: `dbGet(ctx.db, path)` / `dbWaitField(...)` / `arrayContains`
- เซิร์ฟเวอร์: `getIdToken(email)` → โทเค็น ID ของเพอร์โซนา แล้ว
  `apiCall(method, path, { token, body })`

ห่อแต่ละการตรวจสอบไว้ใน `reporter.step(device, 'name', async () => { … })` — มัน
จับเวลาขั้นตอน, ถ่ายภาพหน้าจอ, บันทึกผ่าน/ไม่ผ่าน และเมื่อล้มเหลวจะจับ
testTags บนหน้าจอ เพิ่มอ็อบเจ็กต์ใหม่ลงในอาร์เรย์ `all` ใน `buildJourneys`

ตรรกะล้วน (การแยกวิเคราะห์, ตัวเลือก, การจัดการอาร์กิวเมนต์) ถูกทดสอบหน่วยใน
`tests/scripts/device-journey-runner.test.js` (`cd express-api && npm test`);
ชั้นอุปกรณ์/Firestore/API ถูกทดสอบการรวมโดยการรันชุดทดสอบบน
อุปกรณ์จริง
