# مُشغّل اختبارات الرحلات على الجهاز لتطبيق ShyTalk

_هذه ترجمة لملف JOURNEY-RUNNER.md._

يقوم `device-journey-runner.js` بتشغيل **تطبيق ShyTalk الحقيقي على هاتف متّصل**
عبر رحلات مستخدم شاملة من البداية إلى النهاية، ويكتب **تقريرًا تفصيليًا بالنجاح/الفشل**
يمكنك قراءته — فتُشغّل أمرًا واحدًا وتقرأ تقريرًا واحدًا بدلًا من النقر
على كل خطوة يدويًا.

إنه مُشغّل **هجين**. يمكن لكل رحلة أن تتحقّق من ثلاث طبقات في آنٍ واحد:

1. **واجهة المستخدم** — ينقر على التطبيق الحي ويفحصه عبر `adb` + `uiautomator` (تظهر
   `testTag`s الخاصة بـ Compose على شكل `resource-id`s في المخرجات؛ وتُطابَق مربّعات الحوار
   عبر نصّها المرئي).
2. **Firestore** — يقرأ المحاكي المحلي مباشرةً (عبر `firebase-admin`) للتأكيد على
   حالة قاعدة البيانات خلف كل إجراء.
3. **الخادم / الـ API** — يسجّل الدخول بصفة كل شخصية (رمز هوية Firebase حقيقي من
   محاكي Auth) ويستدعي `express-api`، فيتحقّق بذلك من **القواعد التي يفرضها
   الخادم** (بوابة الفئة العمرية الخاصة بـ OSA، وتجاوز المشرف، والإشراف) — وهي
   _غير_ مرئية في واجهة المستخدم وحدها.

> توجد ترجمات هذا الدليل في `journey-runner-locales/` (20 لغة).

---

## 1. المتطلّبات المسبقة

| تحتاج إلى                  | كيف                                                                                                                                                   |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| تشغيل **Docker Desktop**   | لمحاكيات Firebase + LiveKit/MinIO                                                                                                                     |
| **رفع الحزمة المحلية**     | `bash local/start.sh` (من جذر المستودع) — يُشغّل محاكيات Firebase + express-api. اتركها تعمل.                                                         |
| **زرع الشخصيات**           | `cd express-api && node --env-file=.env.local scripts/seed-personas-local.js` (متكرّر بأمان؛ يزرع طاقم الاختبار P‑02…P‑19 بكلمة المرور `localdev123`) |
| **هاتف متّصل**             | يجب أن يسرد `adb devices` هاتفًا واحدًا (كابل USB **أو** `adb` لاسلكي). ويعمل محاكي Android أيضًا.                                                    |
| **Java 21+ و Android SDK** | مطلوبان في المرة الأولى فقط، حتى يتمكّن المُشغّل من بناء التطبيق إن كان ملف APK مفقودًا                                                               |

يبني المُشغّل بنفسه ملف APK لإصدار التصحيح `local` إن لم يكن مبنيًّا بالفعل.

## 2. تشغيله

من جذر المستودع:

```sh
# تشغيل المجموعة بالكامل مقابل الحزمة المحلية
node express-api/scripts/device-journey-runner.js

# عرض قائمة الرحلات دون تشغيل أي شيء
node express-api/scripts/device-journey-runner.js --list

# تشغيل رحلات محدّدة فقط
node express-api/scripts/device-journey-runner.js --journeys J02,J08,J11

# فرض بناء APK جديد أولًا
node express-api/scripts/device-journey-runner.js --rebuild

# قائمة الخيارات الكاملة
node express-api/scripts/device-journey-runner.js --help
```

الخيارات: `--target local|dev` (الافتراضي `local`) · `--serial <adb-serial>`
(الافتراضي: اختيار تلقائي) · `--journeys <ids>` · `--rebuild` · `--no-reset` (يتخطّى
إعادة التثبيت النظيفة في رحلة الـ smoke) · `--out <dir>` · `--list` · `--help`.

يثبّت المُشغّل serial واحدًا لـ adb لكل أمر، فيعمل حتى عندما يظهر
الهاتف مرّتين (USB + لاسلكي). وللهدف `local` يُعدّ أنفاق
`adb reverse` كي يصل التطبيق على الجهاز إلى الحزمة على جهازك.

## 3. الاطّلاع على النتائج

عند الانتهاء يطبع ملخّصًا ويكتب، ضمن `journey-results/`:

| الملف                           | ما هو                                                                                                           |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `latest-report.md`              | **اقرأ هذا** — لكل رحلة ولكل خطوة ✅/❌ مع السبب، وعلامات testTags الظاهرة على الشاشة، ورابط لقطة شاشة لكل خطوة |
| `latest-report.json`            | البيانات نفسها، بصيغة مقروءة آليًا                                                                              |
| `runs/<runId>/*.png`            | لقطة شاشة لكل خطوة (الناجحة _و_ الفاشلة)                                                                        |
| `runs/<runId>/report.{md,json}` | التقرير المؤرشف لتلك التشغيلة المحدّدة                                                                          |

رمز الخروج هو `0` عند نجاح كل رحلة، و`1` عند فشل أيّ منها. وعند حدوث فشل
تسجّل الخطوة بدقّة ما كان معروضًا على الشاشة، فيمكنك معرفة _السبب_ دون
إعادة تشغيل الهاتف.

## 4. ما الذي تغطّيه الرحلات

شغّل `--list` للاطّلاع على المجموعة الحالية. وباختصار، تغطّي المجموعة:

- **Smoke** — تثبيت نظيف ← القبول القانوني ← تسجيل الدخول، والخلفية قابلة للوصول.
- **تسجيل الدخول حسب الفئة العمرية** — تسجّل شخصيات البالغ / القاصر / المشرف الدخول عبر
  مُنتقي شخصيات dev داخل التطبيق؛ وتُؤكَّد الهوية مقابل طبقة التصحيح المتراكبة وحقل
  `cohort` في Firestore.
- **بوابة الفئة العمرية الخاصة بـ OSA** — لا يستطيع القاصر متابعة شخص بالغ أو الاطّلاع عليه (يُرجِع الخادم
  `404`، ولا تحدث الكتابة في Firestore أبدًا)، بينما تنجح الإجراءات ضمن الفئة العمرية نفسها
  — ما يُثبت أن البوابة خاصة بالفئة العمرية، وليست حظرًا شاملًا.
- **المشرف** — تجاوز الفئة العمرية مقصور على الموظفين (يُرفض عضو عادي بالرمز
  `422`؛ وينجح حساب موظف ويكتب صفّ تدقيق تنظيميًا).
- **الإشراف** — بلاغ ← تعليق من المشرف (+ تدقيق) ← طعن ← رفع التعليق، مفروضةً بالكامل
  من جانب الخادم، مع تنظيف متكرّر بأمان.

تستخدم المصادقة في الرحلات دائمًا **مُنتقي شخصيات dev داخل التطبيق** — وليس أبدًا
تسجيل الدخول الحقيقي عبر Google/Apple.

> **ملاحظة حول مواصفات الرحلات.** خطط Gherkin الموجودة في
> `.project/test-plans/manual/j01-j19` _طموحة_ جزئيًا: فهي تشير إلى
> واجهة مستخدم لا يملكها التطبيق المُطلَق (مثل شاشة تسجيل بالبريد الإلكتروني/كلمة المرور، وعلامات تبويب
> مخفية للقاصرين، وشاشة اكتشاف). لذا يربط المُشغّل النيّة الحقيقية لكل رحلة
> مقابل التطبيق **الفعلي** + Firestore + الـ API، ويسجّل تلك
> الفروقات بوصفها findings بدلًا من الفشل بسبب أمور غير قائمة.

## 5. استكشاف الأخطاء وإصلاحها

| العَرَض                                                      | الحل                                                                                                       |
| ------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| `No adb device found`                                        | وصّل الهاتف / أقرنه؛ وتحقّق من `adb devices`.                                                              |
| تعلّق عند الوصول إلى SignIn / "backend NOT reachable"        | الحزمة المحلية ليست مرفوعة أو لم تُضبط أنفاق `adb reverse` — أعد تشغيل `bash local/start.sh` وأعد التشغيل. |
| `persona "<email>" not found in picker`                      | الشخصيات غير مزروعة — شغّل أمر الزرع في §1.                                                                |
| غياب `Firestore assertions: ON` / تخطّي خطوات قاعدة البيانات | تحقّقات قاعدة البيانات تعمل فقط مع `--target local`.                                                       |
| فشل بناء APK                                                 | افتح ملف `gradle-build.log` المطبوع؛ وتأكّد من تثبيت Java 21+ و Android SDK.                               |
| فشل خطوة على شاشة لم تتوقّعها                                | افتح لقطة الشاشة المذكورة في `latest-report.md` لتلك الخطوة.                                               |

## 6. إضافة رحلة

الرحلات كائنات بسيطة تملك دالة `run(device, reporter, ctx)`، تتكوّن
من المساعدات المشتركة:

- `signInAs(device, reporter, ctx, email, nameToken)` — يسجّل دخول شخصية عبر
  المُنتقي ويتجاوز الشاشات الفاصلة عند الإطلاق الأول وصولًا إلى Home.
- واجهة المستخدم: `tapId` / `waitForId` / `waitForText` / `selectPersonaByText` /
  `tapLowestText`، و`dump(device)` + `byId` / `byText` / `byTextContains`.
- Firestore: `dbGet(ctx.db, path)` / `dbWaitField(...)` / `arrayContains`.
- الخادم: `getIdToken(email)` ← رمز هوية شخصية، ثم
  `apiCall(method, path, { token, body })`.

غلّف كل تحقّق داخل `reporter.step(device, 'name', async () => { … })` — فهو
يقيس زمن الخطوة، ويلتقط لها لقطة شاشة، ويسجّل النجاح/الفشل، وعند الفشل يلتقط
علامات testTags الظاهرة على الشاشة. أضف الكائن الجديد إلى مصفوفة `all` في `buildJourneys`.

أمّا المنطق الصِّرف (التحليل، والمُحدِّدات، ومعالجة الوسائط) فيُختبر وحدويًا في
`tests/scripts/device-journey-runner.test.js` (`cd express-api && npm test`)؛
وتُختبر طبقات الجهاز/Firestore/الـ API تكامليًا بتشغيل المجموعة على
جهاز حقيقي.
