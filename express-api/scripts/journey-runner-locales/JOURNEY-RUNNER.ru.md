# Исполнитель пользовательских сценариев ShyTalk на устройстве

_Это перевод JOURNEY-RUNNER.md._

`device-journey-runner.js` прогоняет **реальное приложение ShyTalk на подключённом телефоне**
через сквозные пользовательские сценарии и записывает **подробный отчёт об успехах/сбоях**, который вы
можете прочитать — так вы запускаете одну команду и читаете один отчёт вместо того, чтобы
вручную проходить каждый шаг нажатиями.

Это **гибридный** исполнитель. Каждый сценарий может проверять три слоя одновременно:

1. **UI** — нажимает/инспектирует живое приложение через `adb` + `uiautomator` (`testTag`-и Compose
   отображаются как `resource-id`-ы в дампе; диалоги сопоставляются по
   их видимому тексту).
2. **Firestore** — читает локальный эмулятор напрямую (через `firebase-admin`), чтобы
   подтвердить состояние базы данных, стоящее за каждым действием.
3. **Сервер / API** — входит как каждая персона (реальный токен ID Firebase из
   эмулятора Auth) и вызывает `express-api`, тем самым проверяя **правила, которые
   обеспечивает сервер** (когортный барьер OSA, переопределение администратором, модерация) — которые
   _не_ видны в одном лишь UI.

> Переводы этого руководства находятся в `journey-runner-locales/` (20 языков).

---

## 1. Предварительные требования

| Что нужно                     | Как                                                                                                                                                      |
| ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Запущенный **Docker Desktop** | для эмуляторов Firebase + LiveKit/MinIO                                                                                                                  |
| **Поднятый локальный стек**   | `bash local/start.sh` (из корня репозитория) — запускает эмуляторы Firebase + express-api. Оставьте его работающим.                                      |
| **Засеянные персоны**         | `cd express-api && node --env-file=.env.local scripts/seed-personas-local.js` (идемпотентно; засевает тестовый состав P‑02…P‑19 с паролем `localdev123`) |
| **Подключённый телефон**      | `adb devices` должен выводить один (USB-кабель **или** беспроводной `adb`). Эмулятор Android также подходит.                                             |
| **Java 21+ и Android SDK**    | нужны только в первый раз, чтобы исполнитель мог собрать приложение, если APK отсутствует                                                                |

Исполнитель сам собирает отладочный APK `local`, если он ещё не собран.

---

## 2. Запустите его

Из корня репозитория:

```sh
# Run the whole suite against the local stack
node express-api/scripts/device-journey-runner.js

# See the list of journeys without running anything
node express-api/scripts/device-journey-runner.js --list

# Run only specific journeys
node express-api/scripts/device-journey-runner.js --journeys J02,J08,J11

# Force a fresh APK build first
node express-api/scripts/device-journey-runner.js --rebuild

# Full option list
node express-api/scripts/device-journey-runner.js --help
```

Опции: `--target local|dev` (по умолчанию `local`) · `--serial <adb-serial>`
(по умолчанию: автовыбор) · `--journeys <ids>` · `--rebuild` · `--no-reset` (пропустить
чистую переустановку в smoke-сценарии) · `--out <dir>` · `--list` · `--help`.

Исполнитель закрепляет **один** серийный номер adb для каждой команды, поэтому он работает даже когда
телефон отображается дважды (USB + беспроводной). Для цели `local` он настраивает
туннели `adb reverse`, чтобы приложение на устройстве достигало стека на вашей машине.

## 3. Посмотрите результаты

Когда он завершает, он печатает сводку и записывает в `journey-results/`:

| Файл                            | Что                                                                                                                      |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| `latest-report.md`              | **Читайте этот** — по сценарию, по шагу ✅/❌ с причиной, экранными testTags и ссылкой на снимок экрана для каждого шага |
| `latest-report.json`            | те же данные, машиночитаемые                                                                                             |
| `runs/<runId>/*.png`            | снимок экрана каждого шага (как успешного, _так и_ сбойного)                                                             |
| `runs/<runId>/report.{md,json}` | архивный отчёт для этого конкретного прогона                                                                             |

Код выхода — `0`, когда все сценарии прошли, `1`, когда хоть один сбойнул. При сбое
шаг фиксирует ровно то, что было на экране, поэтому вы можете увидеть _почему_, не
прогоняя телефон заново.

## 4. Что покрывают сценарии

Запустите `--list` для актуального набора. Вкратце набор покрывает:

- **Smoke** — чистая установка → принятие юридических условий → вход, бэкенд доступен.
- **Когортный вход** — персоны взрослый / несовершеннолетний / админ входят через
  встроенный dev-выбор персон; личность подтверждается по отладочной накладке и
  полю `cohort` в Firestore.
- **Когортный барьер OSA** — несовершеннолетний не может подписаться на взрослого или просмотреть его (сервер возвращает
  `404`, а запись в Firestore так и не происходит), тогда как действия в пределах одной когорты
  успешны — что доказывает, что барьер специфичен для когорты, а не сплошная блокировка.
- **Админ** — переопределение когорты доступно только персоналу (обычный участник отклоняется с
  `422`; учётная запись персонала успешна и записывает строку нормативного аудита).
- **Модерация** — жалоба → приостановка администратором (+ аудит) → апелляция → снятие приостановки, полностью
  обеспечено сервером, с идемпотентной очисткой.

Аутентификация в сценариях всегда использует **встроенный dev-выбор персон** — никогда
реальный вход через Google/Apple.

> **Примечание о спецификациях сценариев.** Планы на Gherkin в
> `.project/test-plans/manual/j01-j19` отчасти _идеализированы_: они ссылаются на
> UI, которого нет в выпущенном приложении (например, экран регистрации по email/паролю, скрытые
> вкладки несовершеннолетних, экран обзора). Поэтому исполнитель сопоставляет реальное намерение каждого сценария
> с **фактическим** приложением + Firestore + API и фиксирует такие
> расхождения как находки, а не сбоит на вымысле.

## 5. Устранение неполадок

| Симптом                                                    | Исправление                                                                                                                 |
| ---------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `No adb device found`                                      | Подключите / спарьте телефон; проверьте `adb devices`.                                                                      |
| Застряло на достижении SignIn / "backend NOT reachable"    | Локальный стек не поднят или туннели `adb reverse` не настроились — перезапустите `bash local/start.sh` и запустите заново. |
| `persona "<email>" not found in picker`                    | Персоны не засеяны — запустите команду seed из §1.                                                                          |
| Отсутствует `Firestore assertions: ON` / шаги БД пропущены | Проверки БД выполняются только для `--target local`.                                                                        |
| Сборка APK падает                                          | Откройте напечатанный `gradle-build.log`; убедитесь, что установлены Java 21+ и Android SDK.                                |
| Шаг сбоит на экране, которого вы не ожидали                | Откройте снимок экрана, названный в `latest-report.md` для этого шага.                                                      |

## 6. Добавление сценария

Сценарии — это обычные объекты с методом `run(device, reporter, ctx)`, составленные
из общих помощников:

- `signInAs(device, reporter, ctx, email, nameToken)` — входит персоной через
  выбор персон и проходит вставки первого запуска до Home.
- UI: `tapId` / `waitForId` / `waitForText` / `selectPersonaByText` /
  `tapLowestText`, и `dump(device)` + `byId` / `byText` / `byTextContains`.
- Firestore: `dbGet(ctx.db, path)` / `dbWaitField(...)` / `arrayContains`.
- Сервер: `getIdToken(email)` → токен ID персоны, затем
  `apiCall(method, path, { token, body })`.

Оберните каждую проверку в `reporter.step(device, 'name', async () => { … })` — это
замеряет время шага, делает его снимок экрана, фиксирует успех/сбой и при сбое захватывает
экранные testTags. Добавьте новый объект в массив `all` в `buildJourneys`.

Чистая логика (парсинг, селекторы, обработка аргументов) покрыта модульными тестами в
`tests/scripts/device-journey-runner.test.js` (`cd express-api && npm test`);
слои устройства/Firestore/API покрыты интеграционными тестами через прогон набора на
реальном устройстве.
