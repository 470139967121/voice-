# Запускач наскрізних тестів ShyTalk на пристрої

_Це переклад файлу JOURNEY-RUNNER.md._

`device-journey-runner.js` проводить **справжній застосунок ShyTalk на під'єднаному
телефоні** через наскрізні користувацькі сценарії та записує **детальний звіт про
успіх/невдачу**, який ви можете прочитати — отже, ви запускаєте одну команду й
читаєте один звіт замість того, щоб вручну проходити кожен крок дотиками.

Це **гібридний** запускач. Кожен сценарій може перевіряти три рівні одночасно:

1. **UI** — торкається/перевіряє живий застосунок через `adb` + `uiautomator`
   (`testTag` у Compose з'являються як `resource-id` у дампі; діалоги
   зіставляються за їхнім видимим текстом).
2. **Firestore** — читає локальний емулятор безпосередньо (через `firebase-admin`),
   щоб підтвердити стан бази даних за кожною дією.
3. **Сервер / API** — входить як кожна персона (справжній токен Firebase ID з
   емулятора Auth) і викликає `express-api`, тож перевіряє **правила, які
   застосовує сервер** (когортний шлюз OSA, перевизначення адміністратором,
   модерацію) — які _не_ видно лише в UI.

> Переклади цього посібника розташовані в `journey-runner-locales/` (20 мов).

---

## 1. Передумови

| Що потрібно                  | Як                                                                                                                                                     |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Запущений **Docker Desktop** | для емуляторів Firebase + LiveKit/MinIO                                                                                                                |
| **Піднятий локальний стек**  | `bash local/start.sh` (з кореня репозиторію) — запускає емулятори Firebase + express-api. Залиште його працювати.                                      |
| **Засіяні персони**          | `cd express-api && node --env-file=.env.local scripts/seed-personas-local.js` (ідемпотентно; засіває тестовий склад P‑02…P‑19 з паролем `localdev123`) |
| **Під'єднаний телефон**      | `adb devices` має показувати один (кабель USB **або** бездротовий `adb`). Емулятор Android також підходить.                                            |
| **Java 21+ та Android SDK**  | потрібні лише першого разу, щоб запускач міг зібрати застосунок, якщо APK відсутній                                                                    |

Запускач сам збирає налагоджувальний APK `local`, якщо його ще не зібрано.

---

## 2. Запустіть його

З кореня репозиторію:

```sh
# Run the whole suite against the local stack
node express-api/scripts/device-journey-runner.js

# See the list of journeys without running anything
node express-api/scripts/device-journey-runner.js --list

# Run only specific journeys
node express-api/scripts/device-journey-runner.js --journeys J02,J08,J11

# Force a fresh APK build first
node express-api/scripts/device-journey-runner.js --rebuild

# Full option list
node express-api/scripts/device-journey-runner.js --help
```

Опції: `--target local|dev` (за замовчуванням `local`) · `--serial <adb-serial>`
(за замовчуванням: автовибір) · `--journeys <ids>` · `--rebuild` · `--no-reset`
(пропустити чисте перевстановлення в smoke-сценарії) · `--out <dir>` · `--list` ·
`--help`.

Запускач закріплює **один** adb-серійний номер для кожної команди, тож він працює
навіть тоді, коли телефон з'являється двічі (USB + бездротовий). Для цілі `local`
він налаштовує тунелі `adb reverse`, щоб застосунок на пристрої досягав стека на
вашій машині.

## 3. Перегляньте результати

Коли він завершується, то друкує підсумок і записує в `journey-results/`:

| Файл                            | Що                                                                                                                                        |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `latest-report.md`              | **Читайте це** — по кожному сценарію, по кожному кроку ✅/❌ із причиною, екранні testTag та посилання на знімок екрана для кожного кроку |
| `latest-report.json`            | ті самі дані, придатні для машинного зчитування                                                                                           |
| `runs/<runId>/*.png`            | знімок екрана кожного кроку (як успішного, _так і_ невдалого)                                                                             |
| `runs/<runId>/report.{md,json}` | заархівований звіт для цього конкретного запуску                                                                                          |

Код виходу — `0`, коли всі сценарії пройшли, і `1`, коли будь-який зазнав невдачі.
У разі невдачі крок записує, що саме було на екрані, тож ви можете побачити
_чому_ без повторного керування телефоном.

## 4. Що охоплюють сценарії

Запустіть `--list`, щоб побачити актуальний набір. Загалом набір охоплює:

- **Smoke** — чиста інсталяція → прийняття юридичних умов → вхід, бекенд досяжний.
- **Вхід за когортою** — персони дорослого / неповнолітнього / адміністратора
  входять через вбудований у застосунок селектор персон для розробників;
  особистість підтверджується щодо налагоджувального накладання та поля
  `cohort` у Firestore.
- **Когортний шлюз OSA** — неповнолітній не може підписатися на дорослого чи
  переглянути його (сервер повертає `404`, а запис у Firestore так і не
  відбувається), тоді як дії в межах тієї самої когорти успішні — що доводить,
  що шлюз специфічний для когорти, а не суцільне блокування.
- **Адміністратор** — перевизначення когорти доступне лише персоналу (звичайного
  учасника відхиляють із `422`; обліковий запис персоналу успішний і записує
  регуляторний рядок аудиту).
- **Модерація** — поскаржитися → адміністратор призупиняє (+ аудит) → апеляція →
  зняти призупинення, повністю забезпечується сервером, з ідемпотентним
  прибиранням.

Автентифікація у сценаріях завжди використовує **вбудований у застосунок селектор
персон для розробників** — ніколи справжній вхід через Google/Apple.

> **Примітка щодо специфікацій сценаріїв.** Плани Gherkin у
> `journey-tests/j01-j19` частково _бажані_: вони посилаються на UI,
> якого немає у випущеному застосунку (наприклад, екран реєстрації за
> електронною поштою/паролем, приховані вкладки для неповнолітніх, екран
> дискаверингу). Тому запускач зіставляє справжній намір кожного сценарію з
> **реальним** застосунком + Firestore + API і записує такі розбіжності як
> знахідки, замість того щоб зазнавати невдачі на вигадках.

## 5. Усунення несправностей

| Симптом                                                    | Виправлення                                                                                                                  |
| ---------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| `No adb device found`                                      | Під'єднайте / спаруйте телефон; перевірте `adb devices`.                                                                     |
| Застрягання на досягненні SignIn / "backend NOT reachable" | Локальний стек не піднято або тунелі `adb reverse` не налаштувалися — перезапустіть `bash local/start.sh` і запустіть знову. |
| `persona "<email>" not found in picker`                    | Персони не засіяні — запустіть команду засівання з §1.                                                                       |
| Відсутнє `Firestore assertions: ON` / кроки з БД пропущено | Перевірки БД виконуються лише для `--target local`.                                                                          |
| Збірка APK зазнає невдачі                                  | Відкрийте надрукований `gradle-build.log`; переконайтеся, що встановлено Java 21+ та Android SDK.                            |
| Крок зазнає невдачі на екрані, якого ви не очікували       | Відкрийте знімок екрана, названий у `latest-report.md` для цього кроку.                                                      |

---

## 6. Додавання сценарію

Сценарії — це звичайні об'єкти з методом `run(device, reporter, ctx)`, скомпоновані
зі спільних помічників:

- `signInAs(device, reporter, ctx, email, nameToken)` — увійти як персона через
  селектор і пройти інтерстиціали першого запуску до Home.
- UI: `tapId` / `waitForId` / `waitForText` / `selectPersonaByText` /
  `tapLowestText`, а також `dump(device)` + `byId` / `byText` / `byTextContains`.
- Firestore: `dbGet(ctx.db, path)` / `dbWaitField(...)` / `arrayContains`.
- Сервер: `getIdToken(email)` → токен ID персони, потім
  `apiCall(method, path, { token, body })`.

Обгорніть кожну перевірку в `reporter.step(device, 'name', async () => { … })` — це
заміряє час кроку, робить його знімок екрана, записує успіх/невдачу й у разі
невдачі захоплює екранні testTag. Додайте новий об'єкт до масиву `all` у
`buildJourneys`.

Чиста логіка (парсинг, селектори, обробка аргументів) тестується модульно в
`tests/scripts/device-journey-runner.test.js` (`cd express-api && npm test`);
рівні пристрою/Firestore/API проходять інтеграційне тестування шляхом запуску
набору на справжньому пристрої.
