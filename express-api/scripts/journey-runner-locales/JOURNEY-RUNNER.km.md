# កម្មវិធីដំណើរការតេស្តដំណើរ (journey-test) លើឧបករណ៍ ShyTalk

_នេះជាការបកប្រែនៃ JOURNEY-RUNNER.md។_

`device-journey-runner.js` បញ្ជា **កម្មវិធី ShyTalk ពិតប្រាកដនៅលើទូរស័ព្ទដែលបានភ្ជាប់**
ឲ្យឆ្លងកាត់ដំណើររបស់អ្នកប្រើពីដើមដល់ចប់ (end-to-end) ហើយសរសេរ **របាយការណ៍ ជាប់/ធ្លាក់ លម្អិត** ដែលអ្នក
អាចអានបាន — ដូច្នេះអ្នកដំណើរការពាក្យបញ្ជាមួយ ហើយអានរបាយការណ៍មួយ ជំនួសឲ្យការចុច
ឆ្លងកាត់ជំហានទាំងអស់ដោយដៃ។

វាជាកម្មវិធីដំណើរការ **ច្របាល់ (hybrid)។** ដំណើរនីមួយៗអាចផ្ទៀងផ្ទាត់លើបីស្រទាប់ក្នុងពេលតែមួយ៖

1. **UI** — ចុច/ត្រួតពិនិត្យកម្មវិធីផ្ទាល់តាមរយៈ `adb` + `uiautomator` (Compose
   `testTag` បង្ហាញជា `resource-id` ក្នុង dump; ប្រអប់សន្ទនា (dialogs) ត្រូវបានផ្គូផ្គងតាម
   អត្ថបទដែលមើលឃើញរបស់វា)។
2. **Firestore** — អានពី emulator មូលដ្ឋានដោយផ្ទាល់ (តាមរយៈ `firebase-admin`) ដើម្បី
   បញ្ជាក់ស្ថានភាពមូលដ្ឋានទិន្នន័យនៅពីក្រោយសកម្មភាពនីមួយៗ។
3. **ម៉ាស៊ីនមេ / API** — ចូលគណនីជា persona នីមួយៗ (ID token ពិតប្រាកដរបស់ Firebase ពី
   Auth emulator) ហើយហៅ `express-api`, ដូច្នេះវាផ្ទៀងផ្ទាត់ **ច្បាប់ដែលម៉ាស៊ីនមេ
   អនុវត្ត** (ច្រកត្រួតពិនិត្យ cohort របស់ OSA, ការ override ដោយ admin, ការសម្របសម្រួល) — ដែល
   _មិន_ អាចមើលឃើញតែក្នុង UI ឡើយ។

> ការបកប្រែនៃមគ្គុទេសក៍នេះមាននៅក្នុង `journey-runner-locales/` (២០ ភាសា)។

## 2. ដំណើរការវា

ពី root របស់ repo៖

```sh
# ដំណើរការសំណុំទាំងមូលប្រឆាំងនឹង stack មូលដ្ឋាន
node express-api/scripts/device-journey-runner.js

# មើលបញ្ជីដំណើរ ដោយមិនដំណើរការអ្វីទាំងអស់
node express-api/scripts/device-journey-runner.js --list

# ដំណើរការតែដំណើរជាក់លាក់
node express-api/scripts/device-journey-runner.js --journeys J02,J08,J11

# បង្ខំឲ្យបង្កើត APK ថ្មីជាមុនសិន
node express-api/scripts/device-journey-runner.js --rebuild

# បញ្ជីជម្រើសពេញលេញ
node express-api/scripts/device-journey-runner.js --help
```

ជម្រើស៖ `--target local|dev` (លំនាំដើម `local`) · `--serial <adb-serial>`
(លំនាំដើម៖ ជ្រើសរើសស្វ័យប្រវត្តិ) · `--journeys <ids>` · `--rebuild` · `--no-reset` (រំលង
ការដំឡើងឡើងវិញស្អាតក្នុងដំណើរ smoke) · `--out <dir>` · `--list` · `--help`។

កម្មវិធីដំណើរការនឹងចាក់សោ adb serial **មួយ** សម្រាប់រាល់ពាក្យបញ្ជា ដូច្នេះវាដំណើរការ សូម្បីតែពេល
ទូរស័ព្ទបង្ហាញពីរដង (USB + ឥតខ្សែ)។ សម្រាប់គោលដៅ `local` វារៀបចំ
ឧម៉ោង `adb reverse` ដើម្បីឲ្យកម្មវិធីលើឧបករណ៍ឈានដល់ stack នៅលើម៉ាស៊ីនរបស់អ្នក។

## 3. មើលលទ្ធផល

នៅពេលវាបញ្ចប់ វាបោះពុម្ពសេចក្តីសង្ខេប ហើយសរសេរ ក្រោម `journey-results/`៖

| ឯកសារ                           | អ្វី                                                                                                              |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `latest-report.md`              | **អានវានេះ** — តាមដំណើរ តាមជំហាន ✅/❌ ជាមួយមូលហេតុ, testTags លើអេក្រង់, និងតំណភ្ជាប់រូបថតអេក្រង់សម្រាប់រាល់ជំហាន |
| `latest-report.json`            | ទិន្នន័យដូចគ្នា ដែលម៉ាស៊ីនអាចអានបាន                                                                               |
| `runs/<runId>/*.png`            | រូបថតអេក្រង់នៃរាល់ជំហាន (ទាំងជាប់ _និង_ ធ្លាក់)                                                                   |
| `runs/<runId>/report.{md,json}` | របាយការណ៍ដែលបានទុកក្នុងបណ្ណសារសម្រាប់ការដំណើរការជាក់លាក់នោះ                                                       |

លេខកូដចេញ (exit code) គឺ `0` នៅពេលដំណើរទាំងអស់ជាប់, `1` នៅពេលណាមួយធ្លាក់។ ពេលធ្លាក់
ជំហាននោះកត់ត្រាយ៉ាងពិតប្រាកដនូវអ្វីដែលនៅលើអេក្រង់ ដូច្នេះអ្នកអាចមើលឃើញ _ហេតុអ្វី_ ដោយមិនចាំបាច់
បញ្ជាទូរស័ព្ទម្តងទៀត។

## 4. អ្វីដែលដំណើរទាំងនេះគ្របដណ្តប់

ដំណើរការ `--list` ដើម្បីមើលសំណុំផ្ទាល់។ មើលដោយរួម សំណុំនេះគ្របដណ្តប់៖

- **Smoke** — ដំឡើងស្អាត → ការទទួលយកផ្នែកច្បាប់ → ការចូលគណនី, backend ឈានដល់បាន។
- **ការចូលគណនី Cohort** — persona មនុស្សពេញវ័យ / អនីតិជន / admin ចូលគណនីតាមរយៈ
  កម្មវិធីជ្រើស persona សម្រាប់អ្នកអភិវឌ្ឍ (dev persona picker) ក្នុងកម្មវិធី; អត្តសញ្ញាណត្រូវបានបញ្ជាក់ប្រឆាំងនឹង debug overlay និង
  វាល Firestore `cohort`។
- **ច្រកត្រួតពិនិត្យ cohort របស់ OSA** — អនីតិជនមិនអាច follow ឬ មើលមនុស្សពេញវ័យបានឡើយ (ម៉ាស៊ីនមេឆ្លើយតប
  `404`, ហើយការសរសេរទៅ Firestore មិនដែលកើតឡើង), ខណៈពេលដែលសកម្មភាពក្នុង cohort ដូចគ្នា
  ជោគជ័យ — ដោយបង្ហាញថាច្រកនេះជាក់លាក់តាម cohort មិនមែនជាការទប់ស្កាត់ទាំងស្រុង។
- **Admin** — ការ override cohort គឺសម្រាប់តែបុគ្គលិក (សមាជិកធម្មតាត្រូវបានបដិសេធជាមួយ
  `422`; គណនីបុគ្គលិកជោគជ័យ ហើយសរសេរជួរ audit តាមបទប្បញ្ញត្តិ)។
- **ការសម្របសម្រួល** — report → admin suspend (+ audit) → appeal → unsuspend, ត្រូវបាន
  អនុវត្តដោយម៉ាស៊ីនមេទាំងស្រុង, ជាមួយការសម្អាត idempotent។

ការផ្ទៀងផ្ទាត់អត្តសញ្ញាណក្នុងដំណើរតែងតែប្រើ **dev persona picker ក្នុងកម្មវិធី** — មិនដែលប្រើ
ការចូលគណនី Google/Apple ពិតប្រាកដឡើយ។

> **កំណត់សម្គាល់អំពី specs របស់ដំណើរ។** ផែនការ Gherkin ក្នុង
> `journey-tests/j01-j19` គឺមួយផ្នែក _ជាក្តីសង្ឃឹម (aspirational)_៖ វាយោងទៅ
> UI ដែលកម្មវិធីដែលបានចេញផ្សាយមិនមាន (ឧ. អេក្រង់ចុះឈ្មោះតាម email/password, ផ្ទាំង (tab) អនីតិជនលាក់,
> អេក្រង់ស្វែងរក (discovery))។ ដូច្នេះកម្មវិធីដំណើរការ map បំណងពិតប្រាកដនៃដំណើរនីមួយៗ
> ប្រឆាំងនឹងកម្មវិធី **ពិតប្រាកដ** + Firestore + API ហើយកត់ត្រាភាពខុសគ្នាបែបនេះ
> ជាការរកឃើញ ជំនួសឲ្យការធ្លាក់លើរឿងប្រឌិត។
