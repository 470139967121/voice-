# SHY-0038: Full bidirectional sync — SHY `.md` files → `public/roadmap-data.json` auto-regen + GitHub Project board link on the public roadmap page

## User Story

As **(a)** a ShyTalk maintainer and **(b)** an outside contributor / observer of the project, I want **the public `shytalk.com/roadmap` webpage to be a derived view of the SHY `.md` corpus (auto-regenerated by CI whenever a SHY status flips) AND to link to the canonical ShyTalk Stories GitHub Project board** so that the two operator-mandated surfaces (public webpage + GH Project board) stay in lockstep with the `.md` source of truth, never drift, and never require a human to remember to update them.

## Why

[[feedback-stories-epics-and-two-surface-sync]] HARD GLOBAL rule (operator 2026-06-07 ~20:48 BST): "every SHY status flip MUST sync to BOTH the public roadmap webpage on shytalk.com AND the GitHub Project board on the repo page; sync MUST be automated (CI regen webpage on merge; SHY-0002 mirror updates Project board); never manual-edit the surfaces."

Today: the SHY-0002 mirror handles the GH Project board side. The webpage side is hand-maintained — a status flip in a SHY `.md` does NOT propagate to `public/roadmap-data.json`. That breaks the rule's _automated_ clause and exposes us to silent drift.

This SHY closes the webpage half of the sync, plus adds the missing cross-link from the webpage to the GH Project board. It absorbs the previously-reserved SHY-0039 (whose entire scope was "CI workflow for SHY → roadmap-data.json auto-sync") because the operator-explicit prioritisation of **quality + reliability + free cost over speed** (2026-06-08 ~18:38 BST) makes the two-PR split a quality compromise: between SHY-0038 ship and SHY-0039 ship the rule would be half-true, and the schema fields added by a smaller SHY-0038 would be dead-code until SHY-0039 catches up.

**Scope alternatives explored (recorded for architect/operator review):**

- **Option A — Link only.** Footer link to GH Project. No JSON change. Pros: zero regression risk; instant. Cons: doesn't deliver the sync rule; forces a schema migration next SHY.
- **Option B — Link + dead schema.** A + optional `epicId`/`shyId` JSON fields with no consumer. Pros: small impl. Cons: schema fields rot until SHY-0042 grouping UI lands; doesn't deliver the sync rule. _(Was the prior recommendation when scope was speed-optimised; rejected under the new quality-first criterion.)_
- **Option C — Link + schema + EPIC grouping UI.** B + render EPIC badges + grouping toggle. Pros: visible EPIC concept. Cons: doesn't deliver the sync rule; visual regression risk on the existing layout.
- **Option D — Full bidirectional sync (CHOSEN).** A + JSON schema extension + CI workflow that regenerates `roadmap-data.json` from SHY `.md` frontmatter on every push-to-main that touches `.project/stories/**`. Pros: literal rule delivery; single source of truth; no manual sync drift possible. Cons: bigger PR (~800-1000 LOC); requires "what's public?" filter rule design.
- **Option E — Live GH Issues API at page-load.** No JSON file; webpage fetches on render. Pros: always fresh. Cons: anonymous GH API rate limit (60 req/hr/IP) breaks the page at scale; depends on GH API uptime; rejected on reliability grounds.
- **Option F — Hybrid live + cached.** E + D's static JSON as fallback. Pros: highest theoretical reliability. Cons: doubles test surface; YAGNI — D's auto-regen already keeps JSON ~30s fresh after any push, so the live path adds complexity for negligible freshness gain.
- **Option G — GH-native README/wiki.** Auto-generated wiki page; shytalk.com just links to it. Pros: zero JS complexity. Cons: visitors leave shytalk.com to see the roadmap detail; SEO hit; abandons the polished website aesthetic.

Operator decision (2026-06-08 ~18:38 BST): Option D, optimising for quality + reliability + free cost over speed/effort.

**Two Day-0 design decisions resolved by the operator (same timestamp):**

1. **Filter rule** = opt-in via `public: true` SHY frontmatter (default = internal). Safer default — a new SHY can't accidentally leak internal content.
2. **Auto-commit loop guard** = `if: github.actor != 'github-actions[bot]'` on the sync workflow. Most robust — the workflow refuses to re-run on its own commit; no commit-message magic; clean audit trail.

### Error paths

- [ ] If a SHY `.md` has `public: true` but is missing required public-facing fields (e.g. no `title`, no `description` derivable from User Story), the sync script MUST fail loudly with a per-file error path + line + missing-field name; CI MUST exit non-zero; no partial JSON write.
- [ ] If `public/roadmap-data.json` becomes malformed (e.g. by a manual edit that the sync script then mis-handles), the existing webpage MUST still render the `_headers` skeleton/error state — page does not white-screen.
- [ ] If the GitHub API is unreachable during the sync workflow (rare; sync doesn't call API directly but Node setup might), the workflow fails the run, does NOT commit a partial JSON, and surfaces the failure on the PR/main check.
- [ ] If a SHY has `public: true` AND `status: Cancelled`, the script MUST emit it with `status: Cancelled` (not silently drop it) — public visibility is independent of cancellation.
- [ ] If TWO SHYs share the same `roadmap_ids` entry (e.g. both claim `G009`), the script MUST fail with a duplicate-claim error naming both file paths — prevents silent merge of distinct work into one roadmap item.

### Edge cases

- [ ] An EPIC with zero public-surfaced children: still emitted in `_meta.epicCount` but contributes nothing to phase items.
- [ ] A SHY with `public: true` but no `epic:` field: surfaced as a top-level item, no EPIC grouping (renderer treats as orphan; that's fine — EPIC grouping UI is SHY-0042's scope).
- [ ] A SHY flips `public: true` → absent (was public, now internal): next sync run REMOVES it from JSON. Renderer must not crash on an item disappearing between renders (browser refresh is the user's path).
- [ ] An EPIC file has `status: Cancelled`: its child SHYs still surface based on their own `public:` flag (an epic being cancelled doesn't auto-cancel the work).
- [ ] First-run on a corpus with zero `public: true` SHYs: script writes a JSON with empty `shyId` arrays in phases — does NOT crash; CI commits the (mostly-empty) `_meta` block as a baseline.
- [ ] Concurrent push race: two pushes to main land within seconds. Second sync run sees JSON already regenerated by the first; `git diff --quiet` returns no diff; no second commit. No infinite loop.

## Risks & Mitigations

- **Risk: auto-commit loop despite the actor guard.** Some workflow triggers (e.g. `workflow_run`) ignore `github.actor`. Mitigation: pin the trigger to `on: push` only, never `workflow_run`; include the path filter `.project/stories/**` so JSON-only commits don't even attempt; AC has an explicit test that the guard is present.
- **Risk: filter rule wrong on first pass.** A SHY accidentally marked `public: true` could leak internal context to visitors. Mitigation: opt-in default (vs. opt-out) is the safer failure mode — accidentally-private is a missing-info bug, accidentally-public is a leak. Only SHY-0038 is opted-in this round; broader triage is its own SHY.
- **Risk: deterministic-sort breakage** ⇒ noisy diffs on every regen even when nothing semantic changed ⇒ commit churn. Mitigation: sort key = (phase index, priority alpha, created ISO date, SHY ID) — stable across runs. Unit test verifies determinism (run script twice, assert byte-identical output).
- **Risk: ~30s sync lag confuses contributors.** A SHY flips Done but the webpage still shows In Progress for ~30-90s. Mitigation: document the lag in CLAUDE.md's new "Public-surfacing" subsection; the absolute upper bound (CI runtime) is rare in practice (typical ~30s).
- **Risk: SHY corpus growth bloats JSON.** At 200+ SHYs, JSON could approach 500KB. Mitigation: only `public: true` SHYs are emitted; opt-in is naturally selective; if needed, future SHY adds gzipped serving via `public/_headers` `Content-Encoding`.
- **Risk: workflow needs `contents: write` permission.** Currently most workflows are read-only. Mitigation: scope the permission to just the sync workflow (`permissions: contents: write`); document in the workflow file.
- **Risk: status flip rolled in for SHY-0037 confuses reviewers.** This PR's diff includes SHY-0037's frontmatter flip + INDEX restructure (per established pattern). Mitigation: PR description enumerates the rolled-in block explicitly; commit message has a `chore(status):` co-trailer.
