# SHY-0061: Roadmap renderer reads SHY-derived `phases[].items[]` with shyId badges

## User Story

As a ShyTalk community member viewing shytalk.com/roadmap, I want SHY-derived work items to appear inside their phases with a visible story-ID badge, so that the public roadmap reflects the live story backlog (auto-synced since SHY-0038) instead of only the hand-curated legacy `features[]` list.

## Why

SHY-0038 made `phases[].items[]` + `currentlyWorkingOn` in `public/roadmap-data.json` authoritative, regenerated by CI on every story merge. But `public/js/roadmap-app.js` (939-line vanilla-JS renderer) still consumes only `phases[].features[]` — synced items are invisible to visitors. This is the renderer half the SHY-0038 spec deferred (reserved as SHY-0061 in SHY-INDEX). It also unblocks SHY-0062 (migrating ~95 legacy features into public SHYs) — migration can't start while items don't render.

## Acceptance Criteria

### Happy path
- [ ] Each phase section renders non-in-progress `items[]` entries after the legacy `features[]` entries, using the same row layout (status icon + name) plus a `shyId` badge (e.g. `SHY-0099`) rendered as a visually distinct chip. In-progress items follow the SAME semantics as in-progress features: phase body skips them; they render only in the top In Progress lift (roadmap-app.js:647 skip rule — discovered during RED, tests corrected before implementation).
- [ ] Item rows honour per-item `i18n[currentLang]` name overrides exactly as feature rows do (shared rendering path — no duplicated row-builder).
- [ ] Phase progress percentage and `(done/total)` counts include items: denominator = `features.length + items.length`; numerator includes items with Done status.
- [ ] The GLOBAL donut chart + legend counts (`count-done` / `count-in-progress` / `count-planned`) also include items — intentional (reviewer-surfaced side-effect, adopted): global totals must equal the sum of per-phase math. Pinned by a dedicated legend-count test against the fixture.
- [ ] In-progress items lift into the top "In Progress" section exactly as in-progress features do, retaining their shyId badge. `currentlyWorkingOn` is NOT separately rendered (each entry is by construction a duplicate of an In Progress item — documented in code comment).
- [ ] A phase with `items: []` (most phases today) shows no observable difference vs current output: same phase title, same feature rows, same progress count, zero console errors (asserted functionally via a `page.route` fixture with and without the `items` key — not a byte/snapshot diff).

### Error paths
- [ ] `items` key missing entirely (schemaVersion 1 data / stale CDN cache): renderer treats it as `[]` — no exception, features render as today.
- [ ] Malformed item entry (missing `shyId` or `name`): row skipped defensively; remaining items render; no console error (matches renderer's existing defensive style).

### Edge cases
- [ ] Item status values are NORMALISED before the existing status→icon switch: sync emits title case (`Done`, `In Progress`) while `getStatusIcon` (roadmap-app.js:466) and the in-progress lift (line 529) expect lowercase-hyphenated (`done`, `in-progress`) — verified live in roadmap-data.json (23×`done` vs 1×`Done`, 4×`in-progress` vs 2×`In Progress`). Normalisation = `String(status).toLowerCase().replace(/\s+/g, "-")`, applied in the item path only, with a code comment explaining the casing schism. Unknown status falls back to planned styling, never throws.
- [ ] The phase-skip guard (roadmap-app.js:575 `if (features.length === 0) continue;`) becomes `(features.length + items.length) === 0` so an all-items phase renders (prerequisite for the post-SHY-0062 edge case below).
- [ ] Phase where ALL entries are items (post-SHY-0062 future): progress bar + counts work with `features.length === 0`.
- [ ] Duplicate shyId across phases (sync bug): both render; no de-dup logic (renderer stays dumb; sync owns integrity).

### Performance
- [ ] No additional network requests; rendering stays single-pass over the existing fetched JSON (renderer is already O(n) DOM building).

### Security
- [ ] Item `name`/`shyId` values are inserted via the renderer's existing text-node/escaping path — never `innerHTML` with unescaped data (items are repo-derived but defence-in-depth per renderer convention).

### UX
- [ ] Badge is visually subordinate (small, muted chip) so item names stay scannable; matches existing roadmap chip styling tokens in roadmap.html's CSS.
- [ ] Badge has `aria-label` "Story SHY-NNNN" (translated template) so screen readers announce it meaningfully rather than reading a bare ID.

### i18n
- [ ] New label key `storyBadge` (template like "Story {id}") added to the renderer's `LABELS` map for ALL 20 locales (ar de es fr hi id it ja km ko nl pl pt ru sv th tr uk vi zh + en).
- [ ] `express-api/tests/scripts/web-i18n-coverage.test.js` extended with a PER-LOCALE-BLOCK assertion for `storyBadge`: for each of the 21 locale keys, a regex of the shape `new RegExp('^\\s+' + lang + ':\\s*\\{[^}]*storyBadge', 'ms')` must match roadmap-app.js source — a file-wide substring match is NOT acceptable (it would pass with locales missing).
- [ ] `storyBadge` lives INSIDE each locale's brace block in the LABELS map — NOT added via a late-merged side object like the existing `DISCLAIMER` pattern (roadmap-app.js:333-359), which would sit outside the block the per-locale assertion inspects.

### Observability
- [ ] N/A — static page; failures surface via the existing console-errors-all-pages.spec.ts sweep, which must stay green.

## BDD Scenarios

**Scenario: synced item renders inside its phase with badge**

- **When** the roadmap page renders
- **Then** the Safety & Compliance phase list contains a row "Age-gating per feature" with a chip reading "SHY-0060"
- **And** the row also appears in the top In Progress section with its badge

**Scenario: progress math includes items**
- **Given** a phase with 2 features (1 Done) and 2 items (1 Done)
- **When** the page renders
- **Then** the phase shows "(2/4)" and a 50% progress bar

**Scenario: missing items key is harmless**
- **Given** a phases entry with no `items` key
- **When** the page renders
- **Then** the phase renders identically to the pre-SHY-0061 output and no console errors occur

**Scenario: badge is translated for screen readers**
- **Given** the page locale is Arabic and the fixture contains an item with status `In Progress`
- **When** the page renders
- **Then** the badge chip's aria-label (in both the phase row and the In Progress lift) is the Arabic `storyBadge` template with the ID substituted

**Scenario: all-items phase renders**
- **Given** a phase fixture with `features: []` and one item with status `Done`
- **When** the page renders
- **Then** the phase card is present with the item row and a "(1/1)" count and a 100% progress bar

## Dependencies

## Risks & Mitigations

- **Risk:** progress-bar AND global-donut semantics change once items count (returning visitors see In Progress +1 immediately). **Mitigation:** today only one item exists (SHY-0060); the (x/y) counts make denominators explicit; consistency between phase sums and the donut is preserved by design.
- **Risk:** badge chip styling clashes with the no-rarity-colors constraint. **Mitigation:** neutral theme tokens only, matching existing chips.
- **Risk:** fixture-based Playwright tests drift from the live data shape. **Mitigation:** fixture copied verbatim from the live SHY-0060 item emitted by the sync script (shape pinned by SHY-0038's own tests upstream).
