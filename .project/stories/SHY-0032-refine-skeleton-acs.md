# SHY-0032: Refine the 28 skeleton SHYs + codify no-skeleton rule

## User Story

As the ShyTalk operator who is AFK most of the time, I want every SHY in the backlog to be **born fully refined** — with verifiable AC across all 8 dimensions, BDD scenarios that match those AC, a concrete Test Plan naming real files, and explicit Dependencies / Risks / Out-of-Scope / DoD — so that any resuming Claude session can pick up the highest-priority SHY and start TDD work without needing me to do upstream planning.

## Why

The 28 skeleton SHYs generated by `scripts/convert-roadmap-to-stories.sh` (SHY-0003, merged as PR #1036) each carry `N/A — TBD refinement on pickup` placeholders across all 8 AC dimensions plus stub BDD scenarios and TBD Test Plans. The validator accepts this because exit-13 is presence-based (BDD-scenarios-exist when AC-bullets-exist) — but the operator on 2026-06-07 ~20:15 BST identified this as a workflow defect: skeletons force the resuming session to do planning work before any code can be written, which collides with the "operator approves spec before code" gate and creates a chicken-and-egg deadlock during AFK windows. The new HARD GLOBAL rule is that every SHY is fully refined at creation time, with no `TBD-on-pickup` content allowed. This story refines the existing 28 skeletons as a one-time cleanup and codifies the rule in CLAUDE.md so future SHYs respect it.

This SHY also slots SHY-0024 (NavGraph coexistence) into Tier 1 priority (P0) per the operator's 2026-06-07 ~20:25 BST decision: migrate Android → SharedNavGraph, delete `app/.../NavGraph.kt`. The operator-confirmed evidence: Android's `NavGraph.kt` is missing the `AgeVerificationSubmit` route that `SharedNavGraph.kt` has — a silent feature-parity bug already shipped.

## Risks & Mitigations

- **Risk:** PR diff is very large (~5000-8000 lines across 31 files). Reviewer agent fatigue → missed issues. **Mitigation:** commit-per-tier so the diff is bisectable + reviewable in 5 logical chunks; pre-self-review the validator + prettier before dispatch; brief the reviewer agent with "expect 28 stories' worth of edits in `.project/stories/`; review one file at a time".
- **Risk:** Refinement introduces interpretation errors — e.g. SHY-0029's AC misses an edge case in the ownerFirebaseUid rule tightening. **Mitigation:** for each Tier 1 security story, quote the roadmap's Gap + Fix columns verbatim in the Why section; architect-validate before merge; the implementing pickup gets one more chance to flag during architect's pre-impl pass.
- **Risk:** Parallel SHY work merges into main mid-refinement, creating conflicts on SHY-INDEX.md. **Mitigation:** no other PRs are in flight; if one opens, rebase + re-resort SHY-INDEX deterministically (sort is `priority asc, created asc`).
- **Risk:** The new no-skeleton rule is codified in CLAUDE.md but not enforced by the validator — future Claude could still write skeletons. **Mitigation:** file follow-up SHY for validator tightening (named in Out-of-Scope); update [[feedback-no-skeleton-stories-fully-refined]] memory to point at the validator-tightening SHY once it exists.
- **Risk:** SHY-0024's refinement assumes the `PlatformNavCallbacks` + `PlatformScreens` ports can accommodate every Android-specific call in `NavGraph.kt`. If a callback type is genuinely Android-only (e.g. `ActivityResultContracts.RequestPermission`), the port may need expansion. **Mitigation:** the refined SHY-0024 lists "expand PlatformNavCallbacks if needed" as an explicit AC bullet under `### Error paths`; architect can flag during validation if the existing 13 hooks + 35 screens are insufficient.
